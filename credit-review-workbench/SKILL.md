# credit-review-workbench — 授信审查工作台

## 定位

本 Skill 是**总调度入口**，不把所有细节堆在一起，而是：

1. **识别任务** — 解析目标企业/项目、任务类型、已有文件、期望交付物
2. **创建工作区** — 标准文件夹结构，原始资料不改动
3. **调度子能力** — 按任务类型路由到合适的子 Skill
4. **证据追溯** — 所有关键数字和结论必须有来源，强制使用 `work-analysis-loop` 护栏
5. **统一交付** — Excel 底稿 + 项目融资模型（如适用）+ 中文审查报告 + 资料索引

## 标准工作区结构

收到任务后，第一步创建工作区目录（macOS 路径，Windows 请替换为 D:\\Workbuddy\\）：

```
[工作区根目录]/
└── [企业名称]_[YYYYMMDD]/
    ├── 00-基础资料/                ← 原始文件，禁止修改
    │   ├── 年报_YYYY.pdf
    │   ├── 审计报告_YYYY.pdf
    │   ├── 季报/中报_YYYY.pdf
    │   ├── 评级报告_*.pdf          ← 发债企业
    │   ├── 募集说明书_*.pdf        ← 发债企业
    │   └── 重大事项公告_*.pdf
    ├── 01-财务底稿/                ← 结构化数据和计算
    │   ├── extracted_data.json     ← PDF 提取的原始数据
    │   ├── 财务数据底稿_YYYYMMDD.xlsx
    │   └── DSCR模型_YYYYMMDD.xlsx  ← 如有项目融资测算
    ├── 02-分析报告/                ← 输出报告
    │   ├── [企业名]_审查报告_V1.docx
    │   ├── [企业名]_审查报告_最终版_YYYYMMDD.docx
    │   ├── 验证报告_V1.md
    │   └── 资料索引.md             ← 所有引用来源的清单
    └── 03-历史版本/                ← 归档
```

**创建命令（macOS）：**

```bash
COMPANY="目标企业名"
DATE=$(date +%Y%m%d)
WORK_DIR=~/Documents/Workbuddy/${COMPANY}_${DATE}
mkdir -p "$WORK_DIR"/{00-基础资料,01-财务底稿,02-分析报告,03-历史版本}
echo "工作区已创建：$WORK_DIR"
```

**Windows PowerShell：**

```powershell
$company = "目标企业名"
$date = Get-Date -Format "yyyyMMdd"
$workDir = "D:\Workbuddy\${company}_${date}"
New-Item -ItemType Directory -Path "$workDir\00-基础资料" -Force
New-Item -ItemType Directory -Path "$workDir\01-财务底稿" -Force
New-Item -ItemType Directory -Path "$workDir\02-分析报告" -Force
New-Item -ItemType Directory -Path "$workDir\03-历史版本" -Force
Write-Host "工作区已创建：$workDir"
```

## 核心执行规则

### 规则 1：所有分析步骤在 work-analysis-loop 护栏下执行

每轮只做一个明确步骤，做完输出六元块：
**状态 → 下一步 → 执行 → 验证 → 停止条件 → 交接摘要**

参见 `work-analysis-loop` Skill 的完整说明。

### 规则 2：财务数据覆盖标准

| 数据类型 | 覆盖范围 | 特别要求 |
|---------|---------|---------|
| 年度报告 | 近三个完整会计年度 + 最新已披露年度 | 动态计算，不可硬编码年份 |
| 审计报告 | **近三个完整会计年度**（必须独立审计报告，不可用年报替代） | ⚠️ 严格要求 |
| 近一期数据 | 最新已披露季报/中报/三季报 | 注明"未经审计" |
| 发债资料 | 最新一期评级报告 + 募集说明书 | 优先最新发行批次 |

### 规则 3：合并口径与本部口径分离

- **合并报表**（含子公司）和**本部报表**（母公司独立）必须分开建表
- 合并口径用于整体分析；本部口径用于偿债能力（特别是内部资金上划能力）
- 如报告只有合并口径，必须在报告中明确标注"无本部口径数据"

### 规则 4：缺失数据严禁当作 0

| 缺失情形 | 标注方式 | 绝对禁止 |
|---------|---------|---------|
| 数据未披露 | `[未披露]` | 填 0 / 估算 / 省略 |
| 文件无法获取 | `[无法访问：原因]` | 用其他年度数据替代 |
| 无法核验 | `[无法核验：来源]` | 视作已核验 |
| 不适用 | `[不适用：原因]` | 填 0 |

### 规则 5：先结构化底稿，再写报告

严禁"边写报告边填数据"。正确顺序：
1. 资料采集 → 2. 数据提取 → 3. 结构化底稿（JSON + Excel）→ 4. 核验 → 5. 报告生成

## 全流程执行路径（综合授信审查 G 型）

### Phase 1：任务解析与工作区初始化（Round 1）

1. 输出任务卡，确认四要素
2. 创建标准工作区
3. 判断企业类型（A股/港股/发债/IPO/非上市）
4. 选择执行路径（参考路由表）
5. 拟定分解步骤，列出每轮目标

### Phase 2：资料采集（Round 2-4）

**路由到** `企业公开信息抓取` Skill：
- 获取公司基本信息和数据源标识（orgId / stockId）
- 下载年报、审计报告、季报（巨潮网 / 港交所披露易）
- 发债企业：额外下载评级报告、募集说明书
- 输出：`00-基础资料/` 目录已填充

**验证节点**：
- [ ] 年报数量是否覆盖近三个完整年度？
- [ ] 是否有独立审计报告（非年报附件版）？
- [ ] 近一期季报/中报是否已下载？

### Phase 3：财务数据提取与底稿（Round 5-7）

**路由到** `企业公开信息抓取`（PDF提取模块）：
- pdfplumber 三层架构提取三大报表
- GLM-4V 视觉兜底（缺失>3个科目时触发）
- 合并口径和本部口径分别提取
- 输出：`extracted_data.json` + Excel 财务底稿

**验证节点**：
- [ ] 资产负债表平衡（资产总计=负债+权益）？
- [ ] 现金流量表期末余额=资产负债表货币资金？
- [ ] 跨年数据趋势连续（无异常跳变）？
- [ ] 所有金额单位统一为万元、两位小数？

### Phase 4：关键财务指标计算（Round 8）

在 Excel 底稿中计算五类指标：

| 类别 | 必算指标 |
|------|---------|
| 偿债能力 | 资产负债率、有息负债/EBITDA、流动比率、速动比率、利息保障倍数 |
| 盈利能力 | 毛利率、净利率、ROE（加权）、ROA、EBITDA 利润率 |
| 营运能力 | 总资产周转率、存货周转率、应收账款周转率 |
| 现金流质量 | 经营现金流/净利润、自由现金流、收现比 |
| 成长能力 | 营收增长率、净利润增长率、总资产增长率 |

**如涉及项目融资**，路由到 `现金流测算` Skill 执行 DSCR 建模。

### Phase 5：经营与行业分析（Round 9-10）

- 从年报原文提取主营业务结构、产能产销、竞争格局
- 补充 WebSearch 获取近期重大事项、股东变动
- 如需深度行业研究，路由到 `investment-industry-research` Skill

### Phase 6：报告生成与交付（Round 11-12）

**路由到** `企业公开信息抓取`（V3 报告管道）：
- `report_pipeline.py`（DeepSeek 双模型）→ `generate_report_v3.js`（Node.js 排版）
- 输出 7 表 6 板块中文授信审查报告

**V1 → 验证 → V2 → 对比评估 → 最终版** 四阶段交付。

### Phase 7：资料索引归档（Round 13）

生成 `资料索引.md`，记录所有引用来源：

```markdown
# [企业名称] 授信审查 — 资料索引

## 基础资料
| 文件名 | 类型 | 来源 | 下载日期 | 用途 |
|--------|------|------|---------|------|
| 年报_2024.pdf | 年度报告 | 巨潮网 | 2026-06-28 | 财务数据提取 |
...

## 数据引用来源
| 数据项 | 数值 | 来源文件 | 页码/章节 |
|--------|------|---------|---------|
| 2024年营业收入 | 123,456.78万元 | 年报_2024.pdf | 第52页利润表 |
...
```

## 特殊企业类型处理

### 非上市国有发债主体

代表案例：西安曲江文化产业投资（集团）有限公司

| 特殊项 | 处理方式 |
|--------|---------|
| 无公开年报 | 从债券发行平台（中国货币网/上交所/深交所）获取募集说明书 |
| 审计报告 | 通常附在募集说明书附件中，需单独提取 |
| 评级报告 | 从中国货币网或评级机构官网获取 |
| 合并口径 | 以合并口径为主，本部口径仅在有子公司资产集中时关注 |
| 控股上市子公司 | 需额外从巨潮网获取上市子公司年报，用于交叉验证 |

### 并购贷款目标公司（含重大资产重组）

代表案例：北方长龙（301357）并购贷款

| 特殊项 | 处理方式 |
|--------|---------|
| 还款来源 | 并购目标公司EBITDA / 合并后上市公司经营现金流 |
| DSCR建模 | 路由到 `现金流测算` Skill，使用并购贷款模式（EBITDA替代发电收入）|
| 还款计划 | 按实际约定填写（如四期：26年8月70%/27-29年各10%）|
| 资本金比例 | 资本金与贷款按比例同步支付，影响期初本金 |

### 半导体封测/光通信芯片封装类企业

代表案例：华天科技

| 特殊项 | 处理方式 |
|--------|---------|
| 行业语义层 | 路由到 `semiconductor-packaging-credit-semantic-layer`（如已安装）|
| CFADS/DSCR口径 | 遵循行业语义层约束，使用标准化指标口径 |
| IMA知识库 | 可调用 ima-mcp 进行行业专业数据交叉验证 |

## 质量门控（每阶段完成后必须过检）

### 数据质量门控

```
□ 所有年份覆盖完整（近三个完整年度 + 近一期）
□ 合并口径和本部口径分离
□ 关键科目无缺失（或标注原因）
□ 资产负债表平衡误差 < 1万元
□ 现金科目跨表勾稽差异 < 1万元
□ 所有金额单位为万元，两位小数
□ 缺失数据已标注，未被当作 0 处理
```

### 报告质量门控

```
□ 7 张表格全部生成（产品收入①、地区收入①b、合并BS②、本部BS③、IS④、CF⑤、有息负债⑥、可比企业⑦）
□ 6 大经营板块全部有内容（工商信息、主营业务、经营分析、财务分析、风险因素、行业分析）
□ 每个板块叙述 ≥ 200 字
□ 风险因素 ≥ 5 个编号段落，使用半角括号 (1)(2)(3)
□ 不含"审查结论"章节（由审查员撰写）
□ 资料索引完整，每个数据点可追溯
```

## 停止与交接规范

遇到以下情况，停止执行并向用户报告：

| 情况 | 报告内容 |
|------|---------|
| 关键年报无法下载 | 已获取年份列表、缺失年份、建议替代来源 |
| PDF 扫描件无法提取 | 已尝试方法、建议 OCR 工具或人工录入关键科目 |
| DeepSeek API 失败 | 已完成数据部分、LLM 生成部分待补充 |
| 发现重大数据异常 | 异常描述、可能原因、建议处理方式 |
| 需要用户决策（口径选择等） | 选项列表 + 各选项的影响说明 |

交接摘要格式：

```markdown
## 交接摘要 — [时间戳]

### 已完成
- [步骤列表，含关键产出]

### 进行中（中断点）
- [当前步骤，中断原因]

### 待完成
- [剩余步骤列表]

### 遗留问题
- [问题清单，含建议解决方案]

### 关键文件位置
- 工作区：[路径]
- 最新版报告：[路径]
- 底稿：[路径]
```

## State 持久化（断点恢复）

每轮执行结束后，必须将状态写入工作区 `.state.json`，支持中断恢复。

### 写入时机

| 时机 | 说明 |
|------|------|
| 每轮 Round 结束后 | 验证完成、状态已更新 → 写入 |
| 人工决策后 | 用户确认后立即写入 |
| 触发停止条件时 | 写入最终状态 → 生成 Handoff |

### 读取恢复逻辑

启动 Skill 时：
```
  if .state.json 存在：
    → 读取 state，检查 updated_at
    → 输出："检测到未完成任务，是否继续？"
    → 用户确认 → 从 state.current_step 或 stages_pending[0] 继续执行
    → 用户拒绝 → 备份 .state.json 为 .state.json.archived，从头开始
  else：
    → 从头开始，生成新 task_id
```

### State Schema

使用统一 State Schema（参见 `loop-engineering` 的 `references/state-schema.json`）。


### State 字段（本 Skill 特定扩展）

| 字段 | 类型 | 说明 |
|-------|------|------|
| `task_type` | string | 任务类型（企业授信/项目融资/行业研究） |
| `dispatched_skills` | list[object] | 已调度的子Skill状态 |
| `cross_verification` | object | 跨Skill交叉验证结果 |
| `deliverables_status` | object | 各交付物完成状态 |

### 写入原子性

写入序列：
  1. 将 State 写入 `.state.json.tmp`
  2. `os.replace('.state.json.tmp', '.state.json')`
  3. 确保写入成功（fsync）

## Handoff 格式（停止时生成）

每次触发停止条件（S1-S7）时，必须生成两个文件：

### 1. `.handoff.json`（机器可读）

结构参见 `loop-engineering` Skill 的 Handoff 格式说明。

关键字段：
- `handoff_id`：格式 `HO-MMDD-XXX`
- `stop_reason`：停止原因（S1-S7）
- `state_snapshot`：当前状态快照
- `deliverables`：交付物清单（路径 + 状态）
- `evidence_summary`：证据概况
- `open_issues`：未解决问题列表
- `next_steps`：后续步骤建议

### 2. `HANDOFF.md`（人类可读）

包含：
- 交接ID、时间、停止原因
- 当前状态（阶段、迭代轮次、完成度）
- 交付物清单（表格形式）
- 证据概况（来源数、证据条数、完整度）
- 未解决问题（带建议）
- 后续步骤

### 交接后动作

| 停止原因 | 动作 |
|---------|------|
| S1（完成） | 归档 .state.json，保留 .handoff.json |
| S2/S3/S4/S5 | 保留 .state.json（可恢复），生成 .handoff.json |
| S6（破坏性操作） | 不执行，等待用户显式确认 |
| S7（用户主动停止） | 保留 .state.json（可恢复） |
