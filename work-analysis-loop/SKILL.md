## Trigger（触发条件）

以下情况**必须触发**本 Skill（作为护栏层，通常不独立运行，而是由其他 Skill 调用）：

| 编号 | 触发条件 | 优先级 |
|------|---------|---------|
| T1 | 用户启动任何分析类 Skill（授信审查/行业研究/财务分析/估值/盈利质量） | P0 |
| T2 | 用户说"帮我分析"/"做一份研究"/"分析循环" | P0 |
| T3 | 用户说"每轮做一步"/"带校验的分析"/"需要证据链" | P1 |
| T4 | 用户说"assisted loop"/"可验证的分析" | P1 |

触发后动作：
1. 作为护栏层，自动附加到调用方 Skill 的每轮迭代中
2. 强制执行六元结构（状态 → 下一步 → 执行 → 验证 → 停止条件 → 交接摘要）
3. 强制数据层级标注（[事实]/[抽数]/[计算]/[假设]/[推断]/[观点]）
4. 强制验证清单过检（每轮结束后必须完成 6 项验证）

---

## 核心设计原则

本 Skill 不替代具体的分析能力（财务、行业、授信、估值），而是为所有分析类任务提供**结构化执行框架**，保证：

1. **每轮只做一个明确步骤**，做完再进行下一步
2. **每个关键数字必须有来源**，不可凭空生成
3. **数据层级分离**：事实 / 抽数 / 计算 / 假设 / 推断 / 观点 六层严格区分
4. **缺失即标注**：缺失数据标注为「缺失 / 无法访问 / 未披露 / 无法核验 / 不适用」
5. **有停止条件**：遇到阻塞、权限不足、重复失败、需要人工确认时，立即停止并交接
6. **先结构后结论**：先形成结构化底稿，再写报告结论

## 六元迭代格式（每轮必须包含）

每次执行一个步骤，结束时输出以下六元块：

```
---LOOP ROUND [N]---
【状态】  本轮已完成：[具体完成了什么，含关键数字和来源]
【下一步】下一轮将执行：[明确描述，不超过一个步骤]
【验证】  本轮已核验：[核验了哪些数字/关系/逻辑，结果如何]
【缺口】  发现的缺失/待补充：[具体列举，注明缺失性质]
【停止条件】继续 / [说明] 需停止：[如触发停止，说明原因和交接内容]
【交接摘要】[如停止或交接给人工，写明已完成内容、遗留问题、建议下一步]
---END ROUND [N]---
```

## State 持久化（断点恢复）

每轮 Round 结束后，必须将状态写入工作区 `.state.json`，支持中断恢复。

### 写入时机

| 时机 | 说明 |
|------|------|
| 每轮 Round 结束后 | 验证完成、状态已更新 → 写入 |
| 人工决策后 | 用户确认了某个假设/参数 → 立即写入（不等待本轮结束）|
| 触发停止条件时 | 写入最终状态 → 然后生成 Handoff |

### State 读取逻辑

```
启动 Skill 时：
  if .state.json 存在：
    → 读取 state
    → 检查 updated_at 是否超过 24h（可能过时）
    → 输出："检测到未完成任务 {task_id}，已完成步骤：{completed_steps}，是否从第 {iteration+1} 轮继续？"
    → 用户确认 → 从 pending_steps[0] 继续执行
    → 用户拒绝 → 备份 .state.json 为 .state.json.archived，从头开始
  else：
    → 从头开始，生成新 task_id，创建 .state.json
```

### State Schema

本 Skill 使用统一 State Schema（参见 `loop-engineering` Skill 的 `references/state-schema.json`）。

基础字段已全部使用，无 Skill 特定扩展字段。

### 写入原子性

```
写入序列（防止写入中途崩溃导致文件损坏）：
  1. 将 State 写入 .state.json.tmp
  2. fsync(.state.json.tmp)
  3. os.replace('.state.json.tmp', '.state.json')
  4. fsync(目录)
```

## 数据层级定义（每次引用数据时必须标注层级）

| 层级 | 定义 | 示例 | 处理要求 |
|------|------|------|---------|
| **[事实]** | 来自原始文件的直接引用，未经处理 | "年报第52页：2024年营业收入123,456.78万元" | 标注来源：文件名/页码/章节 |
| **[抽数]** | 从原始来源提取后的结构化数据，未做推导 | "营业收入：2022=X万元，2023=Y万元，2024=Z万元" | 标注提取方式和来源 |
| **[计算]** | 由抽数进行四则运算、财务比率等数值计算 | "收入增速：(Z-Y)/Y×100% = 15.3%" | 标注公式和依赖数据 |
| **[假设]** | 用于填补缺失数据或设定模型参数的前提 | "假设2025Q1收入与2024Q1持平（原因：季报未披露）" | 标注为假设，注明理由 |
| **[推断]** | 基于数据和逻辑的演绎结论，非直接来源 | "毛利率连续下降，推断原材料成本上涨压力加大" | 标注推断依据 |
| **[观点]** | 分析师主观判断、风险评估、建议 | "综合评估，授信风险等级为中等" | 明确标注为主观判断 |

**严禁混用层级。** 不得将推断写成事实，不得将假设隐含在计算中。

## 缺失数据处理规则

| 缺失类型 | 标注方式 | 处理方法 |
|---------|---------|---------|
| 来源文件不存在 | `[缺失：文件不存在]` | 标注并进入补采队列 |
| 文件存在但数据未披露 | `[缺失：未披露]` | 标注，不得推算替代 |
| 无法下载/访问 | `[缺失：无法访问]` | 标注，记录尝试方法 |
| 无法核验真实性 | `[缺失：无法核验]` | 标注，增加风险提示 |
| 该企业该指标本身不适用 | `[不适用：原因]` | 标注并说明 |

**绝对禁止**：将缺失数据当作 0 处理；在报告中用"约"/"估计"掩盖缺失。

## 停止条件（遇到以下情况必须停止并交接）

| 触发条件 | 停止类型 | 交接内容 |
|---------|---------|---------|
| 同一步骤连续3次失败 | 阻塞停止 | 列出失败步骤、尝试方法、建议解决方案 |
| 关键证据材料缺失且无替代来源 | 证据不足停止 | 列出缺失证据、影响范围、是否可继续部分结论 |
| 需要人工权限（下载需登录、数据付费墙） | 权限停止 | 说明所需权限类型，提供自助操作指南 |
| 需要人工确认关键假设 | 确认等待 | 列出待确认项目，说明对结论的影响 |
| 可能产生破坏性操作（覆写原始文件等） | 安全停止 | 明确描述操作，请求显式确认 |
| 用户指令矛盾或不明确 | 歧义停止 | 列出歧义点，提供解释选项 |

| 停止原因 | 动作 |
|---------|------|
| S1（完成） | 归档 .state.json，保留 .handoff.json，删除 .state.json |

## 验证清单（每轮执行后必须过检）

### 数字层面
- [ ] 关键数字是否有来源标注？
- [ ] 单位是否一致（万元/亿元/元，不得混用）？
- [ ] 期间是否一致（会计年度/日历年度，合并口径/本部口径）？
- [ ] 同一数字是否在不同来源出现矛盾？（如有，注明差异）
- [ ] 计算公式是否正确（平衡关系、比率定义）？

### 逻辑层面
- [ ] 推断是否基于充分事实？（充分=至少2个独立来源支持）
- [ ] 结论是否超出数据所能支撑的范围？
- [ ] 是否对相反证据进行了考量？

### 完整性层面
- [ ] 本轮目标是否完全达成？
- [ ] 是否有未记录的缺口？
- [ ] 下一步输入条件是否已准备好？

## 输出分层规范

分析结论必须分层输出，不得混用：

```markdown
## 结论摘要
[观点层：1-3句核心判断，明确标注这是主观判断]

## 事实依据
[事实层 + 抽数层：直接引用，附来源]

## 计算过程
[计算层：公式 + 依赖数据 + 结果]

## 假设说明
[假设层：列出所有假设，说明为什么采用该假设]

## 推断链
[推断层：从事实到结论的逻辑链，每步标注依据]

## 缺口与局限
[列出所有缺失数据和无法核验的内容，及其对结论的影响]
```
